# 一、项目定位与教学目标

## 1.1 定位

以WTO为例，本项目**不是模拟一场 WTO 会议（WTO Meeting）**，而是模拟**成员之间围绕渔业补贴展开的利益博弈（Interest-based Negotiation）**。

学生真正学习的对象不是协定条文本身，而是：

- 为什么不同国家会提出不同立场？
- 为什么这些国家长期坚持自己的观点？
- 为什么有些国家愿意妥协，而有些国家宁可谈判破裂？
- 国家利益、环境保护、经济发展之间是如何平衡的？

**谈判的本质是利益协调，而不是观点辩论。** 国家没有「正确答案」，每个国家都站在自身利益角度谈判。

## 1.2 能力培养目标

国际谈判能力、国家利益分析能力、政策分析能力、联盟构建能力、利益交换（Trade-off）能力。

## 1.3 教学目标

学生完成整个课程后，应具备：

1. 理解 WTO 渔业补贴协定（或 IOTC 相应议题）的背景与核心争议；
2. 能分析不同国家立场形成的经济、政治和社会原因；
3. 能基于国家利益制定谈判目标、红线和利益交换方案；
4. 能在动态环境下进行联盟构建、文本协商和策略调整；
5. 能复盘谈判过程，并与历史谈判实践进行比较分析。

## 1.4 教学三段与平台职责

```
课前（AI 主导能力培养） → 课堂（教师组织，AI 提供秘书处支持） → 课后（AI 复盘，教师点评）
```

**AI 不是来替代老师，而是辅助老师。** 平台在教师原有教学流程基础上提供辅助，不改变课堂组织方式。

|教师课堂流程|平台辅助内容|
|---|---|
|阅读协定案文|讲解案文、解释条款背景（带出处）|
|分析国家立场|提供人工校准的国家画像与历史立场|
|提交谈判方案|协助生成谈判简报草案|
|正式谈判|担任国家知识库助手与谈判秘书处|
|课后总结|生成复盘素材包 / 复盘报告|

# 二、设计原则与系统边界

## 2.1 核心设计思想

> 以**国际谈判议题（Issue）**为核心，以**国际组织原始文件（Evidence）**为证据，以 **AI 抽取 + 人工策展**构建可信知识资产，以**教师创建谈判项目**为组织方式，以**学生围绕议题开展协商一致谈判**为教学活动，以 **AI 全过程辅助和复盘分析**为支撑。

因此：

- PDF 不是知识库；
- 文件不是课程；
- 条款不是模拟对象；
- **Issue（议题）才是整个系统的核心对象**；
- 但**用于教学的 Issue 权威版本来自人工策展，而非抽取结果的直接输出**（见 5.2）。

## 2.2 系统边界

本平台**提供**：知识学习、国家分析、谈判策略建议、谈判记录、复盘分析。

本平台**不负责**：

- 自动替学生完成谈判；
- 自动生成「最终谈判答案」；
- 替代学生进行决策；
- 替代教师进行课堂评价。

AI 仅作为**智库**工具，所有最终决策均由学生自主完成。明确这条边界，教师用起来更放心，也守住了本课「训练学生自己判断」的初衷。

## 2.3 AI 输出可信度三级（防幻觉总纲）

平台所有面向师生的内容，必须标注其可信度等级。**不同等级的内容在界面上必须可区分。**

|等级|名称|生成方式|出处要求|典型内容|
|---|---|---|---|---|
|**L1**|可溯源检索|RAG，答案由原始文件片段支撑|**必须**给出文件编号 + 段落定位|条款解释、历史事件、某国在某次会议的发言|
|**L2**|人工校准资产|AI 起草 → 人工审核 → 发布|标注审核人与审核时间|国家历史立场、教学议题卡、国家画像|
|**L3**|AI 建议草案|模型生成，无事实断言|界面显式标注「AI 建议，仅供参考」|联盟建议、策略建议、谈判简报草案|

**硬性规则：**

- 涉及**具体国家的具体立场、具体数据**的内容，只能是 L1 或 L2，**禁止以 L3 形式呈现**；
- 国家画像中的经济、渔业、补贴数据，必须来自 4.2 的结构化数据源，**不得由模型自由生成**；
- 未达到「已审核（verified）」状态的 L2 资产，不得进入学生可见层。

## 2.4 信息可见性分级

历史谈判的**最终结果**若在准备阶段对学生可见，模拟的博弈空间将当场归零。因此所有知识对象必须带 `visibility` 字段：

|取值|含义|示例|
|---|---|---|
|`student_pre`|谈判前即对学生开放|议题背景、条款解释、各国公开立场倾向|
|`student_post`|复盘阶段解锁|历史最终案文、真实妥协方案、各方实际让步|
|`teacher_only`|仅教师可见|议题的历史落地版本、参考答案区间、评分提示|

**示例**：Fish 1 的 S&DT 议题最终落地为「2 年争端豁免缓冲期」——该结果标记为 `student_post`，仅在复盘时与学生方案对照展示。

# 三、总体架构

## 3.1 三域划分

平台数据分为性质截然不同的两个域，加一个应用层：

```
┌─────────────────────────────────────────────────────────┐
│                  应用域  Application                     │
│   教师端  ·  学生端  ·  AI Agent（4 角色）  ·  管理后台    │
└───────────────┬─────────────────────┬───────────────────┘
                │                     │
    ┌───────────▼──────────┐  ┌───────▼──────────────────┐
    │  知识域 Knowledge     │  │ 运行时域 Simulation      │
    │  （历史·只读·有出处） │  │ （课堂·可写·学生产出）    │
    ├──────────────────────┤  ├──────────────────────────┤
    │ Document / Chunk     │  │ Project                  │
    │ Evidence             │  │ Session                  │
    │ Issue（抽取候选）     │  │ Delegation               │
    │ TeachingIssue（策展） │  │ Speech                   │
    │ Proposal（历史提案）  │  │ Amendment（修正案）       │
    │ CountryPosition      │  │ TextVersion（案文版本）   │
    │ CountryProfile       │  │ Decision                 │
    │ Outcome              │  │ Minutes                  │
    └──────────┬───────────┘  └──────────────────────────┘
               │  课程快照 Snapshot   ▲
               └──────────────────────┘
```

## 3.2 两域的关键差异

||知识域|运行时域|
|---|---|---|
|内容性质|历史事实|学生课堂产出|
|读写|只读，版本化发布|高频读写|
|出处|**必须**有 Evidence 支撑|无需出处，学生创造|
|质量要求|可溯源、可信度分级、人工校准|完整留痕即可|
|生命周期|跨课程长期沉淀|随课程实例创建与归档|

## 3.3 课程快照机制

教师创建谈判项目（Project）时，系统将所选知识域内容**快照**为该 Project 的初始状态：

- 快照后知识域再更新，**不影响进行中的课程**；
- 快照内容包括：教学议题卡、参与国立场卡、国家画像、基线案文（BaseText v0）；
- 快照记录知识库版本号，便于复盘时追溯「学生当时看到的是哪一版」。

**这是两域之间唯一的写入方向**：知识域 → 运行时域。运行时域的学生产出**不回流**知识域（避免学生虚构内容污染史实库）。

### 4.1.3 文档类型路由

「所有 PDF 统一经过 AI 抽取」不成立——不同类型文件的信息密度与信息种类差异极大：

|文档类型|代表|主要可抽取信息|是否含国家立场|
|---|---|---|---|
|提案型|TN/RL/W/xxx|议题、提案内容、提案国、修改理由|是（提案国自身立场）|
|会议纪要型|委员会/谈判组纪要|各国发言、争议焦点|**是（立场的主要来源）**|
|决定型|WT/MIN、WT/L|最终案文、通过情况|否|
|通报型|G/SCM 补贴通报|补贴金额、类型（表格）|否|
|报告型|IOTC Meeting Reports|会议过程、决议通过情况、反对记录|是|
|科学型|Scientific Data、执行摘要|资源状态、评估结论|否|

**抽取管线必须先做类型识别再路由**，不同类型走不同抽取模板。详见《AI知识抽取工作流设计》。

## 4.2 结构化数据源（不走抽取管线）

国家画像中的事实性数据必须来自权威结构化数据源，**不得由模型生成**：

|数据源|用途|形态|
|---|---|---|
|FAO《世界渔业和水产养殖状况》(SOFIA)|捕捞量、资源状态、渔业就业|报告 + 数据表|
|OECD 渔业支持估算 (FSE) 数据库|各国渔业补贴规模与类型|数据库|
|WTO 成员官方补贴通报|补贴通报口径数据|表格型文件|
|世界银行 / IMF|GDP、人均 GDP、人口|数据库|

这类数据需**提前确定并存入知识库，后续在平台手动更新**，并记录数据年份与口径。

## 4.3 人工校准资产（核心教学资产）

这是本平台**教学效果的真正来源**，数量不大但价值最高，采用「AI 起草 + 人工审核」方式生产：

|资产|数量级|内容|可信度|
|---|---|---|---|
|教学议题卡（TeachingIssue）|每场景 5–8 张|议题描述、背景、**方案区间 A/B/C**、争议焦点|L2|
|国家立场卡（CountryPosition）|议题数 × 参与国数（约 90 张）|某国在某议题上的立场、理由、红线倾向、历史演变|L2|
|国家画像（CountryProfile）|每场景 12–18 份|基本国情、渔业情况、补贴情况、国内政治压力|L2（数据部分 L1）|

> **项目关键路径提示**：在本期「检索增强」定位下，项目风险不在 PDF 抽取准确率，而在**这约 100 份人工校准资产的产出工时**。排期应以此为基准。

# 五、知识生产管线（概要）

> 详细设计见《AI知识抽取工作流设计（Extraction Pipeline）》。本章仅定义边界与原则。

## 5.1 本期定位：检索增强（RAG）

知识库的目标是**让 AI 的回答有据可查**，而非重建条款级结构化数据库。因此：

- **做**：文件清洗去重、类型路由、元信息抽取、议题标签、分块与向量化、证据定位（文件编号 + 段落）；
- **不做**：条款级语义结构重建、修正案自动合规校验、提案全量结构化建模。

## 5.2 两条生产线

```
【主线·自动化】
原始文件 → 去重清洗 → 类型路由 → 元信息抽取 → 议题标签 → 分块 → 向量化 → 检索库
                                                        │
【支线·人工策展】                                         │ 作为素材与证据
原始文件 + 结构化数据源 ──→ AI 起草 ──→ 人工审核 ──→ 发布 ←┘
                                        │
                          教学议题卡 / 国家立场卡 / 国家画像
```

**支线是教学模拟的直接输入，主线是支线的证据来源与学生问答的支撑。**

V1 原「五层抽取」中，第三层（提案抽取）、第五层（结果抽取）在本期降级为**策展素材**，不再作为独立的自动化结构化产出。

## 5.3 审核状态机

所有 L2 资产遵循统一状态流转：

```
ai_draft → under_review → human_verified → published
              │
              └→ rejected（退回重写）
```

**未达 `published` 的资产不得进入学生可见层。** 每次状态流转记录操作人与时间。

# 六、知识域对象模型（概要）

> 详细字段设计见《知识库设计（Knowledge Base Design）》。

|对象|说明|关键字段要点|
|---|---|---|
|`Organization`|组织（WTO / IOTC）|决策规则配置、成员类型定义|
|`Instrument`|法律文书（协定 / 决议）|WTO：协定条款；IOTC：Resolution / Recommendation|
|`Meeting`|会议|届次、时间、地点、议程|
|`Document`|原始文件|文件编号、语言、类型、来源 URL|
|`Chunk`|文件分块|文本、段落定位、向量|
|`Evidence`|证据引用|指向 Document + 段落，供 L1 溯源|
|`Issue`|抽取的议题候选|粒度细、数量大、噪声高，**不直接面向教师**|
|`TeachingIssue`|**教学议题（策展）**|方案区间 A/B/C、争议焦点、`visibility`、审核状态|
|`Proposal`|**历史提案**（成员提交的提案文件）|提案国、修改内容、修改理由|
|`CountryPosition`|国家立场卡|议题、国家、立场、理由、红线倾向、历史演变、`visibility`|
|`CountryProfile`|国家画像|基本国情、渔业、补贴、国内政治压力、数据来源与年份|
|`Outcome`|谈判结果|是否采纳、最终案文、`visibility` 默认 `student_post`|

**术语锁定（重要）**：`Proposal` 专指**历史上成员提交的提案文件**（知识域对象）；`Amendment` 专指**学生在模拟中提交的修正案**（运行时域对象）。两者不共用表结构，不混用命名。

# 七、运行时域对象模型

|对象|说明|关键字段|一期状态|
|---|---|---|---|
|`Project`|谈判项目（一门课的一次模拟）|组织、协定、会议、议题集、参与国、决策规则、知识库快照版本|✅|
|`Session`|一次谈判会议|当前步骤（1–8）、开始/结束时间、主席|✅|
|`Delegation`|代表团（学生小组 ↔ 国家）|国家、学生成员、谈判简报、红线声明|✅|
|`Speech`|发言记录|代表团、步骤号、起止时间戳、音频片段、转写文本|✅（一期核心）|
|`Amendment`|**结构化修正案**|见 7.1|⚠️ 一期仅采集，不做引擎处理|
|`TextVersion`|**案文版本**|版本号、父版本、全文、diff、方括号未决段落|❌ 二期|
|`Decision`|决策记录|决策规则、结果、反对方、反对理由|⚠️ 一期人工录入|
|`Minutes`|会议纪要|按 8 步分段的完整记录|✅（自动汇编）|

## 7.1 Amendment 对象设计

8 步流程第 4 步产出的核心对象。**即使一期不做自动整合，也应按此结构采集**，否则二期无历史数据可用。

|字段|说明|
|---|---|
|`amendment_id`|编号|
|`sponsor`|提案国（代表团）|
|`co_sponsors`|联署国|
|`target_locator`|目标定位（条 / 款 / 段）|
|`operation`|操作类型：`Add` / `Delete` / `Modify`|
|`new_text`|拟增或拟改后的文本|
|`rationale`|修改理由（本国利益关联）|
|`linked_issue`|关联的教学议题|
|`status`|`submitted` → `debated` → `accepted` / `rejected` / `bracketed`|

> **边界建议**：一期即便不做案文引擎，也建议用一个**表单**采集上述字段（学生填表提交，系统仅存档）。开发成本极低，但它把学生产出变成了可评分、可复盘的结构化数据。是否纳入一期，请与老师确认。

## 7.2 案文版本管理（二期）

第 6 步「整合修改形成新案文」是本平台最具价值的功能，需要：

- **版本树**：BaseText v0 → v1 → v2 …，每版记录由哪些 Amendment 合并而来；
- **差异视图**：相对上一版的增删改高亮；
- **方括号机制**：未达成一致的表述以 `[方括号]` 保留在案文中，直观展示「还剩多少分歧」——这是真实多边谈判的标准做法，也是极好的教学可视化。

# 八、谈判流程状态机（8 步）

## 8.1 状态机定义

本流程为教师确认的标准流程，WTO 与 IOTC **共用同一状态机**，差异通过参数配置实现。

|序号|步骤|主持|参与者|系统产出对象|退出条件|可配置参数|
|---|---|---|---|---|---|---|
|1|主席宣布会议开始|主席（教师）|全体|`Session` 开启、出席名单|主席宣布|法定人数规则|
|2|介绍本次议题|主席 / AI 秘书处|全体|议题简报、`BaseText v0`|基线案文锁定|议题数量、基线案文来源|
|3|各国代表举手发言|主席|各代表团|`Speech[]`|发言轮结束|发言顺序规则、单次时限|
|4|提交书面修正案|各代表团|各代表团|`Amendment[]`|提交窗口关闭|窗口时长、是否允许联署、每国提交上限|
|5|针对修正案辩论|主席|各代表团|辩论 `Speech[]`、`Amendment.status` 流转|每条修正案表态完毕|辩论轮数、每条时限|
|6|整合修改形成新案文|主席 + AI 秘书处|—|`TextVersion vN` + diff + `[未决]`|主席宣布新案文|是否保留方括号|
|7|协商一致表决通过|主席|全体|`Decision`|**见 8.2**|**`decision_rule`、反对期**|
|8|休会结束|主席|全体|`Minutes`、复盘素材包|—|是否触发日落破裂分析|

**循环规则**：第 3–7 步可循环。若第 7 步未通过，返回第 4 步或第 5 步进入下一轮，直至达成一致或触发截止期。

## 8.2 决策规则参数化（WTO vs IOTC）

第 7 步「协商一致表决通过」在两个组织中的含义**根本不同**，必须参数化：

||**WTO**|**IOTC**|
|---|---|---|
|`decision_rule`|`consensus`|`consensus_then_two_thirds`|
|通过条件|在场成员**无一正式反对**|先争取协商一致；不成则由**出席并投票成员的三分之二多数**通过|
|是否投票|**不投票**|投票|
|生效|通过即生效|秘书通知后 **120 天**生效|
|退出机制|无（一票即可否决）|**反对（objection）即不受该措施约束**；若超过 **1/3** 成员反对，则其他成员亦不受约束|
|决议形态|部长会决定 / 协定条款|Resolution（有约束力）/ Recommendation（无约束力）|

> **教学价值提示**：IOTC 的「反对退出」机制是极佳的教学素材。学生会遭遇一种 WTO 场景下不存在的情形——**提案以 2/3 多数通过了，但主要渔业国集体提出反对，措施对它们不产生约束**。这比 WTO 的一票否决更微妙，能让学生理解「程序上的胜利不等于实质上的成功」。建议在 IOTC 场景中显式设计这一环节。

## 8.3 截止期压力机制

**这是让「宁可破裂」成为有分量的真实选择、而非免费选项的关键装置。**

- **WTO 场景**：引入日落条款倒计时。根据《渔业补贴协定》第 12 条，若在协定生效四年内（即 **2029 年 9 月**前）未通过全面纪律，除非总理事会另有决定，整个协定（含 Fish 1）将终止。
- **课堂映射**：设置显式倒计时，到点仍未达成即判定为**本轮破裂**，触发复盘中的「破裂后果」分析——谈不成不是没有损失，而是大家一起失去协定。
- **IOTC 场景**：对应压力来自资源评估状态与配额分配周期（如某鱼种评估结论恶化、现行措施到期失效）。

## 8.4 一期实现说明

一期**不实现**案文引擎与程序合法性自动判断，但**必须**做到：

1. **录音按 8 步流程打时间戳分段**——一期交付的不是一段完整音频，而是「第 3 步发言 / 第 5 步辩论 / 第 7 步决策」的分段结构化记录；
2. 每段发言标注代表团（国家）；
3. 主席在界面上手动推进步骤，系统记录各步骤起止时间。

**理由**：这三点开发成本极低，却决定了二期的复盘、案文版本管理、行为分析能否基于真实历史数据启动。若一期只留下无结构的录音，二期等于从零开始。

## 9.1 议题数下限的理由

V6 明确：所有议题合并为一份完整协议文本，**不存在分项表决**，只有全体国家同意整套文件，谈判才算成功；学生仅能在若干议题中挑选一到两项划定绝对红线。

这套设计的教学意义在于**跨议题交换**——用「提高补贴透明度」换取「保留远洋捕鱼补贴」。若课程仅选择单一议题，则不存在可交换标的，谈判退化为零和拉锯，评分体系中占 20% 的「利益交换质量」维度直接失效。

**规则**：教师端议题数下限为 3。单议题模式保留，但明确标注为「程序练习模式，不计入利益交换维度评分」。

# 十一、复盘与评价体系

## 11.1 评分标准

按能力维度评分，**各占 20%**：

|维度|看什么|
|---|---|
|立场分析准确度|是否准确把握本国经济/政治/社会利益，立场是否站得住|
|策略与红线|谈判简报是否清晰，红线是否合理并守住|
|联盟与外交|是否有效识别并争取盟友、处理摇摆国|
|利益交换质量|让步是否换来对等收益，而非无谓妥协或一味僵硬|
|结果与复盘|是否达成本国可接受的可行协议（或有理由地选择破裂）；复盘是否深入、能与历史立场对照|

**评分单位为小组（Delegation）**，即以国家代表团为单位评价，不做组内个人区分。因此运行时 `Delegation` 无需成员子表，一期分段录音标注到代表团级别即可。

**AI 评分仅作参考，最终评价由教师给出。**

## 11.2 复盘报告内容（二期）

- **国家目标完成率**：✔ 保留燃油补贴 / ✔ 获得 SDT / ✘ 未能扩大捕鱼配额
- **谈判行为分析**：主动发言、联盟建立、妥协、提出修正案等次数（**仅作过程参考，不直接用于打分**）
- **与历史立场比较**：如历史上印度始终坚持 SDT，本次学生最终放弃 → 指出偏离
- **破裂后果分析**：若触发日落倒计时未达成，分析各方损失
- **改进建议**：如「先建联盟、再提修正案、最后做利益交换」

## 11.3 一期方案：复盘素材包

一期不生成 AI 复盘报告，但**自动汇编一份复盘素材包**供教师人工评分使用（零 AI 成本）：

1. 按 8 步分段的录音与转写文本，逐段标注代表团；
2. 各代表团提交的谈判简报原件；
3. 各代表团提交的修正案原件；
4. 各步骤耗时统计与发言次数统计；
5. 历史对照材料（此时解锁 `student_post` 内容）。

# 十五、扩展路线

本架构可逐步扩展至：WTO Fish 2 / Fish 3 谈判、IOTC 谈判模拟、ICCAT、WCPFC、FAO 国际渔业治理、UNFCCC 气候变化谈判、CBD 生物多样性谈判、UNCLOS 海洋法谈判。

**复用机制**：接入一个新组织只需三步——

2. 建立该组织的**对象映射**（会议 / 文书 / 提案文件的对应关系，参见附录 A）；
