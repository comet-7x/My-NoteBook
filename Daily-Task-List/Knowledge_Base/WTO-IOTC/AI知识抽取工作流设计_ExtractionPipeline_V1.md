# AI知识抽取工作流设计（Extraction Pipeline）V1

> Version：V1.0
> 上游依据：《基于WTO渔业补贴协定的国际谈判智能体设计思路V6》《AI国际谈判教学平台总体设计V3》《知识库设计V2》
> 更新时间：2026-07-22

> **文档范围**：本文档描述从原始文件到可发布知识资产的**生产过程**。
> 目标数据结构见《知识库设计V2》，本文档不重复定义字段。

# 一、设计目标与范围

## 1.1 管线定位

服务于**检索增强（RAG）**定位，目标是两件事：

1. 让 841 份 WTO 文件（及后续 IOTC 语料）**可被精确检索并溯源到段落**；
2. 从语料中**高效产出 270 条人工校准的教学资产**。

## 1.2 输入与输出

| | 内容 |
|---|---|
| **输入** | WTO：841 份英文 PDF；IOTC：筛选后语料（待定）；结构化数据源（FAO SOFIA / OECD FSE / WTO 通报 / World Bank） |
| **输出（自动）** | `Document` / `Chunk` / 向量索引 / `Issue` 候选 / `Proposal` / `PositionSignal`（中间对象） |
| **输出（策展）** | `TeachingIssue` 6 + `IssueOption` 18 + `CountryPosition` 54 + `PositionEvolution` ~190 + `CountryProfile` 9 + `ProfileMetric` ~180 + `Outcome` 6 |

## 1.3 四段总览

```
【第一段 预处理】                【第二段 抽取】
841份PDF                        按 doc_type 路由
  ├─ 编号解析与去重                ├─ 元信息抽取（全类型）
  ├─ 文本层检测 / OCR              ├─ 议题标签
  ├─ 段落切分（para级）            ├─ 提案抽取（proposal）
  ├─ 文档类型识别                  ├─ 立场线索（meeting_report）★
  └─ 质量门禁                      ├─ 结果线索（decision）
        ↓                          └─ 向量化与索引
   Document + Chunk                       ↓
                              Issue候选 / Proposal / PositionSignal
                                          ↓
【第三段 策展】★ 关键路径          【第四段 发布】
  ├─ 教学议题卡     6              ├─ 发布前校验
  ├─ 立场卡矩阵    54              ├─ 可见性复核 ★
  ├─ 演变时间线   ~190             ├─ kb_version 发布
  ├─ 国家画像       9              └─ 增量更新
  └─ 指标导入    ~180
        ↓
   人工审核 → published
```

## 1.4 一个前置判断

在 RAG 定位下，**第一、二段是相对标准的工程，真正决定项目成败的是第三段策展**。

- 第一、二段：841 份文件的处理是可自动化、可并行、可重跑的，出错代价低（重跑即可）；
- 第三段：270 条记录需要人读、人判断、人签字，**不可并行压缩、不可重跑**，且质量直接决定课堂效果。

因此本文档的设计重心是：**第一、二段尽可能为第三段减负**——抽取的目标不是"抽得全"，而是"让策展人员打开工作台就能快速找到该看的那几段原文"。

# 二、第一段：预处理（Preprocessing）

## 2.1 编号解析与多语言归组

### 编号解析

以文件编号（`doc_symbol`）为唯一键，通过规则解析：

| 组织 | 编号形态 | 解析要点 |
|---|---|---|
| WTO | `TN/RL/W/274`、`G/SCM/N/xxx`、`WT/MIN(22)/33`、`WT/L/xxx`、`JOB/RL/xxx` | 前缀段决定机构与文档类型；括号内为年份；末段为流水号；可能带 `/Rev.1`、`/Add.1`、`/Corr.1` 后缀 |
| IOTC | `IOTC-2026-S30-PropXX`、`IOTC-2026-CoC23-R` | 组织-年份-会议-类型-序号 |

**修订版处理规则**：`/Rev.N` 为修订版，`/Add.N` 为增补，`/Corr.N` 为更正。三者与母文件建立 `supersedes` / `supplements` 关系；**检索默认只召回最新修订版**，历史版本保留但降权——否则学生会检索到已被取代的案文。

### 多语言归组

WTO 语料已确认仅保存英文版，该步骤在 WTO 场景空转。**IOTC 官方语言为英语和法语，法文版真实存在，该步骤必须生效**：同编号多语言版本归为一组，英文版置 `is_primary_language=true` 进入检索库。

## 2.2 文本层检测与 OCR

**先检测再处理**，避免对扫描件白跑一遍文本抽取：

```
pdfinfo  → 页数、创建工具
pdffonts → 是否存在字体（即是否有文本层）
   ├─ 有字体 → pdftotext 直接抽取，has_text_layer = true
   └─ 无字体 → 走 OCR 流程，has_text_layer = false
```

**OCR 策略**：页面栅格化后 OCR，语言包按 `language` 字段选择（英/法）。OCR 产出的 Chunk，`para_confidence` 整体上限设为 0.6，并在 Evidence 上标注「定位为近似位置」。

**预期比例**：WTO 官方文件绝大多数为数字化生成、含文本层；OCR 主要出现在早期扫描件与部分附件。实际比例需在样本处理后统计。

## 2.3 段落切分

这是**支撑段落级证据定位的核心步骤**，直接对应知识库 `Chunk.para_label` 与 5.1 节的定位精度要求。

### 切分规则（按优先级）

| 优先级 | 规则 | 适用 | `para_confidence` |
|---|---|---|---|
| 1 | 识别原文显式段落编号（行首 `1.` `12.` `3.2` 等 + 缩进模式） | WTO/IOTC 正式文件普遍带编号 | 1.0 |
| 2 | 编号存在但 OCR 不稳，用编号连续性校验（前后号是否递增） | 扫描件 | 0.7–0.9 |
| 3 | 无编号，按空行 + 缩进 + 句首大写推断 | 会议纪要连续叙述段 | 0.5–0.7 |
| 4 | 表格区域，定位到「页 + 表序 + 行序」，`para_label` 记为 `Table N Row M` | 通报型文件 | 0.6 |

### 长度处理

- **基本单元是原文段落，不做固定长度切分**；
- 超长段落（>1000 字）内部再切，但**保留同一 `para_label`**，通过 `chunk_index` 区分；
- 过短段落（<50 字，如标题行、编号行）与相邻段落合并，但记录原始边界，避免引用时指到一个孤立标题。

### 需特别处理的版式

| 情形 | 处理 |
|---|---|
| 双栏排版 | 先做栏识别再按栏顺序读，否则段落串行 |
| 脚注 | 抽出为独立 Chunk，标记 `section_path = footnote`，不与正文段落混合 |
| 连字符换行 | 合并断词 |
| 页眉页脚 | 识别并剔除，避免污染检索 |
| 表格 | 保留结构化形态（行列），不平铺成散文 |

## 2.4 文档类型识别与路由

**规则优先，LLM 兜底**——文件编号前缀已经包含了绝大部分类型信息，不必为此调用模型。

| 编号前缀 / 特征 | `doc_type` | 抽取路由 |
|---|---|---|
| `TN/RL/W/` | `proposal` | 元信息 + 议题标签 + 提案抽取 |
| `G/SCM/N/` | `notification` | 元信息 + 表格解析（不做立场抽取） |
| `WT/MIN`、`WT/L` | `decision` | 元信息 + 结果线索；**可见性默认 student_post** |
| `WT/GC/M/`、委员会纪要、`IOTC-*-R` | `meeting_report` | 元信息 + 议题标签 + **立场线索抽取** |
| `WT/LET/` | `legal_text` | 元信息 + 条款结构 |
| IOTC `National Report` | `national_report` | 元信息 + 本国数据线索 |
| IOTC 科学文件 / 执行摘要 | `scientific` | 元信息（不做立场抽取） |
| `NGO Statements` | `ngo_statement` | 元信息（**不得作为成员立场证据**） |
| 规则无法判定 | LLM 分类 | 按分类结果路由 |

> **重要约定**：`ngo_statement` 与 `scientific` 类文件**不参与立场线索抽取**。NGO 声明不是成员立场，若混入会直接污染立场卡——这是最容易犯且最难发现的错误之一。

## 2.5 质量门禁

进入第二段前的检查：

| # | 检查 | 未通过处理 |
|---|---|---|
| 1 | `doc_symbol` 解析成功 | 转人工登记 |
| 2 | 文本抽取非空且长度合理（非乱码） | 标 `extraction_status = failed`，转 OCR 或人工 |
| 3 | 段落数 > 0 且 `para_confidence` 均值 ≥ 0.5 | 标记低置信，仍入库但降权 |
| 4 | 语言检测与 `language` 字段一致 | 修正字段 |
| 5 | 页数与 `page_count` 一致 | 重跑 |

**失败文件不阻塞管线**，进入待处理队列，批次结束后统一出报告。

# 三、第二段：抽取（Extraction）

## 3.1 抽取任务清单

| 任务 | 适用 `doc_type` | 产出对象 | 是否入 KB 正式表 |
|---|---|---|---|
| T1 元信息抽取 | 全部 | `Document` 字段补全 | ✅ |
| T2 议题标签 | proposal / meeting_report / decision | `Chunk.issue_tags` | ✅ |
| T3 议题候选抽取 | proposal / meeting_report | `Issue` | ✅（仅供策展，不面向师生） |
| T4 提案抽取 | proposal | `Proposal` | ✅ |
| T5 **立场线索抽取** | meeting_report | `PositionSignal`（中间对象） | ❌ **不入正式表** |
| T6 结果线索抽取 | decision | `OutcomeSignal`（中间对象） | ❌ |
| T7 表格解析 | notification | 表格数据（参考用） | ❌ |
| T8 向量化与索引 | 全部 | 向量库 | ✅ |

## 3.2 提示词设计的四条硬性原则

所有抽取任务的提示词必须遵守：

1. **强制证据绑定**——每一条抽取结果必须携带其来源 `chunk_id`，无来源的结果一律丢弃；
2. **拒绝猜测**——信息不足时返回 `null`，**不允许基于常识补全**。提示词中明确写："若原文未明确说明，返回 null，不要推断";
3. **只做抽取不做归纳**——不要求模型判断"这是否重要"、"这代表什么立场倾向"，只要求它定位并转述原文说了什么；
4. **结构化输出**——严格 JSON，字段与《知识库设计》对齐，解析失败即重试，连续失败转人工。

> 第 2 条是防幻觉的关键。国际组织文件中大量表述是克制的外交语言，模型很容易"帮忙"补全成一个更明确的立场，这正是 V6 反复警告的错误来源。

## 3.3 T5 立场线索抽取（重点）

### 为什么产出的是「线索」而不是「立场卡」

V6 明确要求国家历史立场**必须人工校准，不可由 AI 自由生成**。因此：

- T5 的产出是 `PositionSignal`——**某国在某文件某段落说过什么**，是事实性摘录；
- `CountryPosition` 立场卡——**某国在某议题上的立场及其经济/政治/社会原因**，是判断，由人做出。

**AI 负责把散落在几百份纪要里的相关段落找出来并归类，人负责读完后下判断。** 这是本管线中 AI 与人的分工线。

### `PositionSignal` 中间对象

| 字段 | 说明 |
|---|---|
| `document_id` / `chunk_id` | 来源，必填 |
| `country` | 发言方；**无法确定归属时置 null 并丢弃** |
| `issue_hint` | 疑似关联的议题（自由文本 + 候选 `TeachingIssue` 匹配） |
| `statement_summary` | 原文所述内容的转述（**不照抄，不加评论**） |
| `stance_hint` | 表态倾向：`support` / `oppose` / `neutral` / `conditional` / `unclear`；**默认 `unclear`** |
| `event_date` / `meeting_id` | 时间与场合，供演变时间线使用 |
| `confidence` | 抽取置信度 |

### 关键约束

| 约束 | 理由 |
|---|---|
| 发言方无法确定归属 → 丢弃 | 纪要中常见"一位成员表示"这类匿名表述，归错国比漏掉危害大 |
| `stance_hint` 默认 `unclear` | 迫使模型在证据不足时不表态 |
| NGO 声明、科学文件不参与 T5 | 非成员立场 |
| 集团发言（如"非洲集团"、"ACP 集团"）单独标记 | 不可直接归给某一成员国，但对理解联盟极有价值，供策展参考 |
| 同一段落可产生多条 Signal（多国发言） | 按发言方拆分 |

### 提示词骨架

```
你是国际组织文件的信息抽取助手。以下是一份 {org} {meeting} 会议纪要的一个段落。

任务：找出该段落中【成员方】就渔业补贴相关问题所作的表态。

严格规则：
1. 只抽取原文明确写出的内容，不得推断、不得补全、不得评价。
2. 若无法确定发言方是哪个成员，返回空数组。
3. 若发言方是集团（如非洲集团），group_statement 置 true，不归给单一成员。
4. 表态倾向不明确时，stance_hint 一律填 "unclear"。
5. statement_summary 用你自己的话转述，不要照抄原文句子。
6. 输出严格 JSON，无其他文字。

段落原文（chunk_id={chunk_id}）：
{text}

输出格式：
[{"country": ..., "group_statement": false, "issue_hint": ...,
  "statement_summary": ..., "stance_hint": ..., "confidence": 0.0-1.0}]
```

## 3.4 T3 议题候选与 T4 提案抽取

- **T3 产出 `Issue` 候选，预期 1000+ 条，粒度细、重复多、噪声高**。这是**正常的、可接受的**——它的用途只是让策展人员在工作台上按议题聚类浏览原文，不面向师生。不必投入精力去优化它的去重与合并，那是策展环节的人工工作。
- **T4 `Proposal`** 抽取提案国、目标条款、操作类型、内容概述、理由。`content_summary` 要求转述而非原文照录。

## 3.5 T8 向量化与索引

| 项 | 设计 |
|---|---|
| 嵌入对象 | `Chunk.text` |
| 模型要求 | 需支持英文为主、中文查询（学生用中文提问，语料是英文）→ **必须选择跨语言嵌入模型**，否则中文提问召回率会很差 |
| 索引元数据 | `document_id` / `doc_symbol` / `org` / `meeting` / `year` / `doc_type` / `language` / `issue_tags` / `visibility` / `is_primary_language` |
| 过滤前置 | 检索时先按元数据过滤再做向量召回 |

> **`visibility` 必须进入向量索引的元数据**，否则学生检索时会绕过可见性控制，直接从 `decision` 类文件中读到历史结果。这是《知识库设计》9.2 中列出的头号风险，堵点在这里。

## 3.6 批处理与成本

### 估算方法

精确成本需在样本上标定，此处给出方法与量级：

```
总 token ≈ Σ(每份文件字符数) ÷ 平均字符/token × (1 + 输出比)
待抽取文件数 = 841 × 参与 T2–T6 的类型占比
```

**需要先做的事**：抽 30–50 份代表性文件（覆盖各 `doc_type`）跑一遍完整管线，标定三个数——平均页数、平均 token 数、单份处理耗时与成本。**在此之前的任何总成本数字都是猜测**，不应写入排期。

### 工程要求

| 项 | 要求 |
|---|---|
| 幂等 | 以 `document_id + task_id` 为键，重跑不产生重复数据 |
| 断点续跑 | 按批次记录进度，中断后从断点继续 |
| 失败重试 | JSON 解析失败重试 2 次，仍失败转人工队列 |
| 并发控制 | 按 API 速率限制配置并发，避免限流 |
| 批次报告 | 每批输出：成功数、失败数、低置信数、各任务产出量 |
| 可回滚 | 每批数据带 `batch_id`，可整批撤回重跑 |

### 分批建议

按**文档类型**而非文件顺序分批，优先处理对策展价值最高的类型：

| 批次 | 内容 | 优先级 | 理由 |
|---|---|---|---|
| B1 | `meeting_report` | **最高** | 立场线索的唯一来源，直接喂给 54 张立场卡 |
| B2 | `proposal`（TN/RL/W 310 份） | 高 | 议题与方案区间的主要素材 |
| B3 | `decision` | 中 | 结果线索，供 `Outcome` 与复盘对照 |
| B4 | `legal_text` | 中 | 基线案文来源 |
| B5 | `notification`、`scientific`、其他 | 低 | 不参与立场抽取，主要作检索补充 |

**B1 完成即可启动策展**，不必等全量抽取结束——这是压缩总工期的关键。

# 四、第三段：策展（Curation）★ 关键路径

## 4.1 策展的地位

**270 条人工校准记录是本项目唯一不可压缩的工作量。** 它不能靠加机器解决，不能重跑，且质量直接决定课堂效果——一张写错的印度立场卡，会让整堂课的学生学到错误的东西。

因此本段的设计目标是：**把人的时间尽可能用在"判断"上，而不是"找资料"上**。

| 环节 | 谁做 | 目标 |
|---|---|---|
| 找出相关原文段落 | AI | 把 841 份文件收敛成每格 5–15 段候选 |
| 起草初稿 | AI | 提供可修改的起点，而非最终答案 |
| **下判断** | **人** | 立场是什么、为什么、经济/政治/社会原因 |
| 挂证据 | 人（AI 辅助） | 每条判断都能指回原文 |
| 审核发布 | 人 | 签字负责 |

## 4.2 策展工作台功能需求

| # | 功能 | 说明 |
|---|---|---|
| 1 | **矩阵视图** | 6 议题 × 9 国 = 54 格，每格显示完成状态与证据数量，一眼看进度 |
| 2 | **证据面板** | 点开某格，左侧自动列出该「国家 × 议题」相关的 `PositionSignal` 与原文段落，按时间排序 |
| 3 | **一键挂证据** | 从证据面板选中段落 → 生成 `Evidence` 并绑定到当前字段 |
| 4 | **AI 起草按钮** | 基于已挂证据生成初稿，写入 `ai_draft` 状态 |
| 5 | **强制约束** | 未挂 `Evidence` 的字段不可提交审核；`rationale_economic`/`rationale_political` 为空不可提交 |
| 6 | **演变时间线编辑器** | 按时间轴增删节点，每节点绑定证据 |
| 7 | **审核视图** | 并排显示 AI 草稿与人工修改，提交后自动写 `ReviewLog` |
| 8 | **原文快速定位** | 点击证据跳转到 PDF 对应页并高亮段落 |
| 9 | **跨国对比** | 同一议题下横向对比 9 国立场，便于发现矛盾与遗漏 |

> 功能 1、2、3 是效率的关键。若缺少证据面板，策展人员要自己在 841 份文件里翻找，工时会成倍上升。

## 4.3 三条策展流水线

```
流水线 A：教学议题卡    6 张 + 18 个方案区间   ← 先做，是 B、C 的前提
流水线 B：国家立场卡   54 张 + ~190 个演变节点  ← 关键路径，最耗时
流水线 C：国家画像      9 份 + ~180 条指标      ← 可与 B 并行
```

**依赖关系**：A 必须先完成，因为 B 的矩阵维度由 A 的议题定义；C 与 B 无依赖，可并行。

## 4.4 流水线 A：教学议题卡

**输入**：V6 的六议题表（已有现成内容）+ `Issue` 候选 + `Proposal` + `legal_text`

**这条流水线的起点比想象中好**——V6 已经给出了 6 个议题、每个议题的 A/B/C 方案区间、以及各国大致立场倾向。策展工作主要是**核实与补全**，而非从零创作。

| 步骤 | 操作 | 产出 |
|---|---|---|
| A1 | 录入 V6 六议题及方案区间作为初稿 | 6 × `TeachingIssue` + 18 × `IssueOption`（`ai_draft`） |
| A2 | 为每个议题挂接背景与争议焦点的证据 | `Evidence` |
| A3 | 从 `legal_text` 提取对应条款作为 `base_text`（模拟基线案文） | `base_text` 字段 |
| A4 | 标注 `is_historical_outcome` 与 `outcome_note`，**确认 visibility 为 student_post** | `IssueOption` 完成 |
| A5 | 建立 `Outcome` 记录 | 6 × `Outcome` |
| A6 | 审核发布 | `published` |

> **V6 待修正项**：V6 标题写"5 项核心议题"，表格实为 6 项（含⑥技术援助机制），正文又写"五项"。已确认以 **6 项**为准，录入时统一。

### 单格 SOP（每格约 30–60 分钟）

|---|---|---|---|
| B1 | **证据召回** | 系统 | 按「国家 + 议题」检索 `PositionSignal` 与 Chunk，按时间排序，返回 5–15 段候选 |
| B2 | **人工筛选** | 人 | 剔除不相关、误归属、集团发言，保留 3–8 段核心证据 |

### 需要特别处理的格

| 情形 | 处理 |
|---|---|
| **俄罗斯 6 格** | V6 四类清单未含俄罗斯，无现成依据。**策展前先做证据可得性检查**：若某议题下 `PositionSignal` 少于 3 条，标记为"证据不足"，与教师商议是改用其他国家还是接受简化立场卡 |
| **证据稀疏格** | 某国在某议题上确实未公开表态是**真实情况**，此时 `stance` 填 `silent`，并在 `rationale` 中说明"未见公开表态"，**不得编造立场** |
| **欧盟** | 作为单一成员方处理，但需注意其内部成员国分歧在纪要中可能单独出现，不混入 |
| **立场随时间反转的格** | `stance_stability` 填 `shifting`，演变时间线必须完整，这类格教学价值最高 |

## 4.6 AI 起草的提示词约定

AI 起草**只能基于人工已筛选的证据**，不得自行检索扩展：

```
以下是【{国家}】在【{议题}】上的相关发言记录，均来自 WTO 官方文件。

请基于且仅基于这些材料，起草一张国家立场卡草稿。

严格规则：
1. 只能使用下方材料中的信息，不得引入任何外部知识。
2. 每一项判断后标注其依据的材料编号 [1][2]。
3. 材料不足以支撑某字段时，该字段填 "证据不足"，不要推测。
4. rationale_economic / political / social 三项若材料中无直接依据，
   填 "证据不足"，由人工补充。不要用常识推断。
5. 这是草稿，将由人工校准，请勿追求完整性，宁缺毋滥。

材料：
[1] {doc_symbol}, Paragraph {para}, {date}：{statement_summary}
[2] ...

输出字段：stance / preferred_option / acceptable_options /
rationale_economic / rationale_political / rationale_social /
red_line_tendency / tradeable / stance_stability
```

> 第 4 条是本项目最容易出问题的地方。模型对"印度为什么支持 S&DT"有大量训练数据里的常识，会写得很流畅很合理——但那不是从本项目语料中来的，无法溯源，也可能已经过时。**流畅的错误比明显的空白危险得多。**

## 4.7 演变时间线的构建

**输入**：该格已筛选的 `PositionSignal`，按 `event_date` 排序

| 步骤 | 操作 |
|---|---|
| E1 | 系统按时间排列该国该议题的全部 Signal |
| E2 | 人工识别关键转折点，通常 3–4 个（起始表态、MC11 前后、MC12、MC12 后） |
| E3 | 为每个节点判定 `change_type`（`initial`/`reinforce`/`soften`/`shift`/`silent`/`withdraw`） |
| E4 | 填写 `driver`（变化动因）——国内选举、资源评估、外部压力等 |
| E5 | 绑定证据 |

**起始年份**：V6 提"2017 年至今"，建议沿用（见附录 B 待确认）。

> `change_type = shift` 与 `soften` 的节点教学价值最高——前者是"为什么会改变"的素材，后者是学生阶段四寻找可交换空间的直接线索。策展时应优先保证这两类节点的证据质量。

## 4.8 流水线 C：国家画像与指标

| 步骤 | 操作 | 数据来源 |
|---|---|---|
| C1 | 从结构化数据源批量导入 `ProfileMetric`（约 180 条） | FAO SOFIA / OECD FSE / WTO 通报 / World Bank |
| C2 | 人工核对数值、单位、年份、口径 | — |
| C3 | AI 起草叙述性字段（基本情况 / 渔业 / 补贴） | 基于 C1 的数值 + 语料 |
| C4 | **人工填写 `domestic_pressure`** | 渔民协会 / 环保组织 / 财政部门 / 地方政府各自的诉求与强度 |
| C5 | 审核发布 | — |

> **C1 的数值禁止 LLM 写入**（《知识库设计》4.13 规则）。若抽取管线从 PDF 中读到数值，只能作为核对参考，不得直接入表。
>
> **C4 是学生阶段二的核心内容**——V6 强调"你代表的是整个国家，而非某一群体"，国内压力的多方博弈正是这句话的载体，必须人工撰写。

## 4.10 质量抽检

发布前按 20% 比例抽检，检查项：

| # | 检查 |
|---|---|
| 1 | 每条 `rationale` 是否真能从所挂证据推出，还是引入了外部常识 |
| 2 | 证据是否指向正确的段落（随机点开验证） |
| 3 | 发言方归属是否正确（尤其区分成员发言与集团发言、NGO 发言） |
| 4 | `stance` 与演变时间线末节点是否一致 |
| 5 | `Outcome` 与 `IssueOption.is_historical_outcome` 的 `visibility` 是否均为 `student_post` |

## 5.1 发布前校验清单

|---|---|---|

| 8 | 同一 `doc_symbol` 下 `is_primary_language=true` 唯一 | ✅ |

## 5.2 可见性复核（专项）

**这是发布前最后一道、也是最容易被忽略的关卡。** 可见性设计再完整，只要有一处漏配，学生就能提前看到历史答案。

| # | 复核项 |
|---|---|
| 1 | 全部 `Outcome` 的 `visibility = student_post` |
| 2 | 全部 `IssueOption.is_historical_outcome` 与 `outcome_note` 字段级受控 |
| 3 | **全部 `doc_type = decision` 的 Document 及其 Chunk 在准备阶段设为 `student_post`** |
| 4 | 向量索引的元数据中 `visibility` 已同步，检索层过滤生效 |
| 5 | **实测**：以学生身份提问"这个议题最后是怎么定的"，确认无法得到答案 |

> 第 5 项是唯一可靠的验证方式。前四项是配置检查，第 5 项是行为检查——**必须实际跑一遍**。
