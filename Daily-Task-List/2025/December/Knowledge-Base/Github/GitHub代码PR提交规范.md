
PR 是团队协作中代码评审、合并的核心环节，规范的 PR 能提升评审效率、降低沟通成本，同时让代码变更可追溯、易维护。以下是通用且可落地的 Git PR 提交规范，可根据团队业务调整。

## 一、核心原则

1. **单一职责**：一个 PR(Pull Request) 仅解决一个问题 / 实现一个功能，避免大而全的 PR（如同时改功能 + 修 bug + 重构），拆分小 PR 更易评审。
2. **可追溯**：PR 信息需关联需求、Bug 单号（如 Jira/Issue），清晰说明 “为什么改、改了什么、怎么验证”。
3. **简洁清晰**：命名和描述避免模糊表述（如 “修复问题”“优化代码”），精准指向改动核心。
4. **自检先行**：提交前完成本地验证、冲突解决、代码规范检查，减少评审者无效反馈。

## 二、PR 命名规范

### 1. 统一格式
```plaintext
[type](scope): 简洁描述（不超过50字符）
```

### 2. 字段说明

| 字段        | 说明                                                                                                                                                                                                                                                                                  |
| --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `type`    | 必选，标识 PR 类型，取值如下：<br><br>✅ feat：新增功能<br><br>✅ fix：修复 Bug<br><br>✅ docs：仅文档修改（如注释、README）<br><br>✅ style：仅格式调整（如缩进、空格，不影响代码逻辑）<br><br>✅ refactor：重构（既不新增功能也不修复 Bug）<br><br>✅ test：补充 / 修改测试代码<br><br>✅ chore：构建 / 工具链 / 依赖调整（如打包配置、CI 脚本）<br><br>✅ revert：回滚之前的提交<br><br>✅ perf：性能优化 |
| `scope`   | 可选，标识改动的模块 / 功能域（如 user、order、pay、api、component），便于定位改动范围                                                                                                                                                                                                                           |
| `subject` | 必选，简洁描述改动核心，用**祈使句**（如 “新增用户登录校验” 而非 “新增了用户登录校验”），结尾不加标点                                                                                                                                                                                                                            |
|           |                                                                                                                                                                                                                                                                                     |

### 3. 示例
```plaintext
feat(user): 新增用户手机号验证码登录
fix(order): 修复订单支付后状态未更新的问题
docs(api): 补充接口参数说明和返回示例
refactor(pay): 优化支付回调参数解析逻辑
test(cart): 为购物车加减商品添加单元测试
chore(build): 升级webpack版本至5.88.0
revert(feat): 回滚“新增优惠券领取功能”
```

## 三、PR 描述规范

描述需结构化、信息完整，便于评审者快速理解背景和改动，模板如下（可根据团队需求调整）：

### 1. 基础模板
```plaintext
### 背景（Why）
- 关联需求/BUG单号：[Jira/Issue链接]
- 改动原因：（如“用户反馈登录后无法记住密码”“需求要求新增订单导出功能”）

### 改动内容（What）
- 核心改动：（如“在登录接口添加remember参数处理逻辑”“新增OrderExportController导出订单接口”）
- 涉及文件：（可选，如“src/user/LoginService.java、src/order/OrderExportController.java”）
- 其他调整：（如依赖新增、配置修改、测试用例补充等）

### 测试验证（How）
- 测试环境：（如“测试服http://test.xxx.com”“本地环境localhost:8080”）
- 测试步骤：（如“1. 登录页面勾选记住密码；2. 退出重新登录，验证密码是否保留”）
- 验证结果：（如“功能正常，参数传递正确，无报错”）

### 注意事项（Attention）
- 兼容问题：（如“需同步更新前端登录表单参数”“影响v1版本接口，需灰度发布”）
- 部署要求：（如“需重启应用”“需执行数据库脚本xxx.sql”）
- 其他：（如“暂未覆盖边缘场景，后续优化”“依赖第三方服务xxx的v2接口”）
```

### 2. 示例
```plaintext
### 背景（Why）
- 关联需求/BUG单号：https://jira.xxx.com/browse/PROJ-1234
- 改动原因：用户反馈订单支付完成后，订单列表仍显示“待支付”，导致用户重复支付

### 改动内容（What）
- 核心改动：修改OrderPayService的paySuccess方法，添加订单状态更新逻辑，确保支付回调后立即更新status为“已支付”
- 涉及文件：src/order/service/OrderPayService.java、src/order/mapper/OrderMapper.xml
- 其他调整：补充OrderPayServiceTest.java中支付成功的单元测试用例

### 测试验证（How）
- 测试环境：测试服 http://test.xxx.com/order
- 测试步骤：
  1. 创建测试订单，选择支付宝支付；
  2. 完成支付后，调用订单列表接口；
  3. 检查订单状态是否为“已支付”。
- 验证结果：订单状态更新及时，接口返回正确，无报错

### 注意事项（Attention）
- 兼容问题：无，仅修改内部逻辑，不影响前端交互
- 部署要求：需重启订单服务
- 其他：已覆盖正常支付场景，异常回调（如重复回调）后续补充幂等处理
```

## 四、提交前自检清单

提交 PR 前必须完成以下检查，避免无效评审：

1. ✅ 代码符合团队编码规范（如命名、注释、格式），通过 lint / 格式化工具校验；
2. ✅ 本地已验证功能正常，无语法错误、运行时异常；
3. ✅ 解决了与目标分支（如 master/develop）的代码冲突；
4. ✅ 提交记录整洁（squash 无关的临时提交，如 “修复拼写错误”“临时调试”）；
5. ✅ 补充了必要的测试用例（单元测试 / 集成测试），且测试用例执行通过；
6. ✅ CI/CD 流水线（如编译、测试、构建）无失败；
7. ✅ 改动范围符合预期，无多余代码（如调试日志、注释掉的代码、无用依赖）。

## 五、PR 审核与协作规范

1. **指定审核人**：根据改动模块选择对应负责人 / 资深开发者作为审核人，避免无目标的 “全员评审”；
2. **及时响应**：评审意见需在合理时间内（如 1 个工作日）回复 / 修改，避免 PR 长期搁置；
3. **小步迭代**：若改动量大（如超过 500 行），拆分多个小 PR，按依赖顺序提交评审；
4. **拒绝 “草稿 PR” 滥用**：仅对未完成的改动标记为 Draft，完成后及时转为 Ready for review；
5. **评审通过后合并**：至少 1 名审核人 Approve 且无未解决的评审意见，方可合并；
6. **合并方式选择**：
    - 功能迭代 / 修复：优先 Squash merge（将多个提交合并为 1 个，保持主干提交整洁）；
    - 分支同步 / 重构：可 Rebase merge（保持提交历史线性）；
    - 避免直接 Merge（易产生冗余合并节点）。

## 六、常见问题避坑

1. ❌ PR 标题模糊：如 “改点东西”“修复 bug”→ 需按规范明确类型和范围；
2. ❌ 描述为空 / 仅贴链接：需补充背景和改动说明，链接仅作为辅助；
3. ❌ 提交未测试的代码：避免 “先提交再测试”，本地验证是基础；
4. ❌ 一次 PR 改多个无关功能：如同时改用户模块和订单模块→ 拆分独立 PR；
5. ❌ 合并后不清理分支：及时删除已合并的功能分支，避免分支泛滥。

## 总结

规范的 PR 提交本质是 “降低协作成本、提升代码质量”，团队可基于以上规范制定适配自身的细则（如补充行业专属的 type/scope），并通过 CI 工具（如 GitHub Actions、GitLab CI）校验 PR 标题格式，强制落地规范。
