## 风险点与待解决的问题：

### 风险点：

1. 算法资源不足：
   1. 现状：算法需完成功能实现、性能优化、效果优化三块。后两者耗时长且客户要求更高，但目前算法仅 2 人。
   2. 影响：能勉强完成功能实现，性能与效果优化大概率没有时间投入，直接影响交付效果。
   3. 建议：增配 1 名算法人员，加快功能实现节奏，为性能/效果优化预留充足时间。
2. 算法侧的测试缺失：
   1. 现状：应用侧已有集成测试，算法侧基本没有测试覆盖。
   2. 影响：直接影响项目质量，问题难以在交付前暴露。
   3. 建议：算法侧不做单元测试，但补齐**集成测试**；建立测试用例库，覆盖边界值测试、功能测试、稳定性测试等，并留档记录。
3. 校方模型 API 上下文受限：
   1. 现状：校方提供的大模型 API 在上下文长度上有限制。
   2. 影响：可能影响检索增强后的回答准确性，进而影响交付质量。
   3. 建议：拿到 API 后尽快实测上下文上限，据此调整 chunk 大小、召回数量等。
4. RAG 数据链路需重构以适配多文件类型：
   1. 现状：现有 RAG 流程为 数据解析 → 数据拆分切片（整页）→ 入库 → 向量检索 → 重排序。校方文件类型种类多，前两环节难以支撑。
   2. 需重构的两部分：
      1. 数据解析：图片、表格处理不够严谨，需按文档类型调整解析策略。
      2. 数据拆分切片：当前整页切片无法保证高质量 chunk 召回，需改为更细粒度的语义切片。
5. 数据包含除开英语和中文的其他语言：
   1. 现状：IOTC部分数据包含法语版本
   2. 影响：可能会影响数据解析，模型回答效果
   3. 建议：提前调研测试相关模型对法语的支持情况
