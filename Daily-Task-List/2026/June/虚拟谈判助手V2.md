

## 产品定位与核心约定

- **产品定位**：沉浸式谈判模拟，核心价值在"过程的真实感"。
- **AI 助手定位**：训练学生的谈判能力，**不代写发言稿**。
- **表决机制**：默认追求协商一致（consensus）；主席可在超时后启用兜底，转为多数表决。
- **立场来源（混合）**：老师在分组时预设每组的**关键红线 + 可量化目标**（作为会后"立场评判"的参考）；其余公开立场、论据、盟友判断由学生自行研究补充。
- **三个核心数据对象**：议题 `Topic`、立场卡 `PositionCard`（公开/私密分层）、决议草案 / 议案 `DraftResolution`（共享、可表决的标的）。

- **知识库（IOTC 语料）是公开、共享的**。National Reports、提案、合规报告在现实中本就是公开发布的，所以学生**可以**查对手国的资料来研究对方，不违反信息不对称。
- **真正私密的是"红线 / BATNA / 量化目标"这层训令**：它不来自语料、不进向量库，只作为本组 AI 助手实例的上下文注入。
- 一句话边界：**对手的资料能查，对手的底牌查不到。** 谈判的张力在后者。

### 角色分类

|角色|职责|关键约束|
|---|---|---|
|老师 / 主席|备课、定议题与议案规则、课前分组指派、指派提案组、主持、推进、表决裁决、查看报告|兼任程序主持与裁判，不替学生做谈判决策|
|学生 / 代表团|代表一个成员国；课前研究、课上发言、组团、表态、投票|只掌握本国训令；他国资料可查、底牌不可查|
|· 提案组（1–N 组）|普通组职责 + **课前起草本次议题的议案草案**|议案课前提交老师审核|
|AI 助手|基于本组训令 + 知识库做问答与追问|实例绑定到"人"、上下文绑定到"组/国"；追问而非代写|
|系统 / 后台|房间与阶段状态机、发言队列、组团、录音、分析|发言由系统调度，故"谁说了什么"天然可锚定|

关于训令的解释：
- **核心目标**:你这个国家在这次议案上最想要什么(比如"保住本国金枪鱼配额不低于历史均值")
- **红线**:绝对不能接受的点(比如"任何把配额砍到 X 以下的方案,宁可否决")
- **可量化指标**:用来事后判断这组谈得好不好(比如"最终文本里本国配额 ≥ 多少吨")
- **可交易筹码 / BATNA（达成谈判协议的最佳选择方案，Best Alternative to Negotiated Agreement）**:愿意拿什么换什么、谈崩了的备选方案

## 阶段 1 · 课前准备

**本阶段目标**：把会议的"事实地基"准备好——议题、议案、分组、各组训令、各组对本国立场的研究。

**老师 / 主席**

- **定议题**：从议题模板库挑选，锚定到一份（或两份）待表决议案。
- **写议案规则**：本版先准备两个议案版本，数量与关系（竞争择一 / 会前定主次 / 可合并）待与老师确认后定稿。议案**会前向所有组公布**。
- **提前分组并指派身份**：每组代表一个成员国；按 40 人左右的班级规模划分（具体国家数 → 组规模待确认）。
- **指派提案组（1–N）**：选定哪些组负责起草议案，及指派方式（抽签 / 老师定 / 自愿，待确认）。
- **预设各组训令**：给每组关键红线 + 可量化目标（混合方案的"老师侧"）。
- **告知检索**：让学生用平台收集本国历年会议相关资料（Meeting Documents、National Reports 等）。

**学生 · 普通组**

- 按"本国 × 文档类型 × 年份"检索本国历年资料，整理本国公开立场、论据、潜在盟友与对手。
- 结合老师给的训令（红线、目标），想清楚哪些可让、哪些不可退。

**学生 · 提案组**

- 在上述研究基础上，**额外起草本次议题的议案草案**，课前提交老师审核。

**平台 / AI 助手**

- 提供按**成员国 + 文档类型 + 年份**的检索（对接语料分库，见附录 A）。
- 系统调用 Steins-RAG 帮学生检索、消化本国资料，帮提案组起草议案——但**不替学生定立场、不代写**。
- AI 助手实例**绑定到人**、上下文**绑定到组/国**：同组 2–3 人各开各的会话，共享同一份本国训令与国家上下文。**每人随时可用**，课前为主，课上若有人没准备好也可随时再准备。

## 阶段 2 · 开场陈述（提案组优先）

**本阶段目标**：先由提案组亮出议案，再由各国亮明公开立场，建立全场"地形图"。

**老师 / 主席**

- **先请提案组陈述议案**及其理由，再依次组织其他国做开场陈述，每国限时。
- 控场，确保每国都有发言机会、不超时。

**学生 / 代表团**

- 提案组：讲清议案内容与本国诉求。
- 其他组：陈述本国公开立场与关切，不必暴露私下底线。
- 听并记录：标注潜在盟友与对手，校正课前的预判。

**AI 助手 / 系统**

- 系统记录每段发言的归属与时间。
- 学生在听别国陈述时，可问助手"这国立场背后真实利益可能是什么"。

---

## 阶段 3 · 自由辩论与组团

**本阶段目标**：围绕议案交锋，逐轮形成团体——这是会议的主体。

**老师 / 主席**

- **主持点名**：其他组举手，主席依次指派发言。
- **维持秩序**：处理程序性动议（答辩权、程序问题），控制单点不过度纠缠。
- **观察团体演化**：留意立场相近的组如何聚成团体、是否有国家被孤立；重复多轮。
- **判断节奏**：观察立场是否收敛，决定何时进入表决。

**学生 / 代表团**

- **举手 → 等待指派 → 发言**：针对议案表态、回应、施压或释放善意。
- **组团**：与立场相近的组协调立场、形成团体或联合表态，争取把议案拉向本国偏好。
- **守住红线**：发言可灵活，但别在公开场合越过本国不可退让点。
- **随时用 AI 助手**：排队、准备时让助手帮组织论点、检索证据。

**AI 助手 / 系统**

- 助手做苏格拉底式追问（"这超出本国某议题的红线了吗""对方刚才碰到了谁的核心利益"），给要点而非成稿。
- 系统持续记录发言归属与录音；支持团体/分组通道（仅参与方可见，可配置）。

> **⚠ 待确认（决定本阶段交互形态）**：辩论过程中**议案是否允许被修正**？
> 
> - 方案甲：只对原议案投赞成/反对（实现简单，回合更短）。
> - 方案乙：可提修正案、改完再投（更接近真实会议，但需要共享草案的逐条编辑与「阶段·起草」环节）。 这两种的"组团"玩法与系统复杂度差别很大，需先定。

---

## 阶段 4 · 表决

**本阶段目标**：对议案形成决议——是否通过。

**老师 / 主席**

- **主持表决**：默认追求协商一致；若长时间僵持触发超时，启用兜底转为多数表决。
- **宣布结果**：通过 / 未通过，以及（若采方案乙）最终文本。

**学生 / 代表团**

- **投票 / 表态**：依据本国训令与谈判结果决定立场。
- 最后权衡：坚守红线宁可否决，还是接受次优方案换取通过。

**AI 助手 / 系统**

- 助手做最后提醒："当前议案是否达成了本国核心目标？否决的代价是什么？"
- 系统记录投票结果。

---

## 阶段 5 · 会后复盘

**本阶段目标**：从过程中提炼收获（沉浸式定位下，v1 分析先做轻，重反馈而非精密打分）。

**老师 / 主席**

- 查看会议报告：群体结果（议案是否通过、与历史真实结果对比、哪些争议点解决/搁置）+ 各代表表现概览。
- 据此点评与评分（维度：参与度、立场忠实度、论证质量、谈判技巧、是否达成本国目标）。

**学生 / 代表团**

- 查看个人复盘报告：发言回顾、是否守住红线、是否达成本国目标，并与历史真实做法 / 建议策略对照。

**AI 助手 / 系统**

- 后台分析链：录音转写 → 用系统发言记录锚定归属 → 逐发言打标（立场/让步/引证）→ 聚合 → 生成两份报告（老师全局版 + 学生个人版）。

## 附录 A · 语料分层与解析取向

按"谈判用途"分层（对接 doc-type-specific chunking）：

|层级|文档类型（数量）|谈判角色|解析 / 切块取向|
|---|---|---|---|
|一层·立场地基|National Reports (348)、CMM Proposals (261)|学生研究本国立场的主源；议案模板与条款级立场光谱|NR 按章节切、强标 country+year、支持按国过滤；Proposals **按条款/article 切并保留结构**|
|二层·战术弹药|Meeting Report (384)、Implementation reports (386)、Compliance Reports (356)、Summary compliance (60)|措施演变/历史投票 = 论据；履约合规 = 筹码|Meeting Report 按议程项切、抽取决议与投票事件；合规类尽量**抽成结构化**（谁对哪项措施达标/未达标）|
|三层·背景|Information papers (1003)、Executive Summaries (301)、Reports from other meetings (279)、NGO Statements (161)|背景理解；NGO 声明 = 外部压力视角（若设观察员角色可用）|常规语义切块即可|
|需先处理|Meeting documents (6018)、None (4)|暂不可直接用|6018 是最大的桶，**极可能混入上面各类的实例**，建议先抽样看构成、跑一遍重分类再入库，否则检索信噪比差|

> 判断：**6018 份"大杂烩"是语料里最大的不确定性。** 若其中藏着大量 NR / Proposal，重分类后"立场地基"层会显著变厚；若多为杂项，则更多是噪声。先抽样定夺。

## 附录 B · 信息可见性规则

|信息|本代表团|其他代表团|主席 / 老师|
|---|---|---|---|
|公开知识库（NR、提案、合规报告等语料）|✅|✅（可查对手资料，真实且允许）|✅|
|私密训令（红线、BATNA、量化目标）|✅|❌|✅（裁判需要）|
|公开立场 / 开场陈述|✅|✅|✅|
|共享议案 / 草案|✅|✅|✅|
|团体内部磋商通道|仅参与方|❌|可旁观（可配置）|

> 私密训令不进共享向量库，只作为本组助手实例的上下文注入——他国助手检索时绝不能命中。

## 附录 C · 待老师确认清单

1. **议案数量与关系**：两个版本——竞争择一 / 会前定主次 / 可合并？
2. **议案是否允许被修正**：只投原案（方案甲）还是可提修正案再投（方案乙）？（决定阶段 3 形态）
3. **参与国数量**：40 人下选几个国家？据此定组规模（如 10 国→每组约 4 人）。
4. **组内分工与发言席位**：是否"一人首席发言、其余研究与磋商"？发言代表能否过程中轮换？
5. **提案组指派**：哪些组、怎么指派（抽签 / 老师定 / 自愿）？
6. **（可选）是否设 NGO 观察员角色**：有 161 份 NGO 声明可作现成素材，增加外部压力与沉浸感。
