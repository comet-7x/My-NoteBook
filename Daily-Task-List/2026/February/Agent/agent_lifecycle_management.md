# Agent 生命周期与会话管理设计笔记

## 1. 问题背景

当前 Agent 实例在创建后被无限期地存储在服务内存的一个普通字典中。随着用户量和会话数的增长，服务内存将持续升高，最终导致内存耗尽（OOM, Out of Memory），服务性能下降甚至崩溃。为了保证服务的稳定性和可扩展性，必须设计一套合理的 Agent 生命周期管理机制。

## 2. 技术选型探讨

我们探讨了以下四种主要方案：

### 方案一：无限期内存字典（现状）

-   **描述**：所有 Agent 实例都保存在一个全局的 Python 字典里，key 为 `session_id`。
-   **优点**：实现简单，对于单用户或测试场景非常直观。
-   **缺点**：存在致命的内存泄漏风险，无法用于生产环境。

### 方案二：LRU (Least Recently Used) 缓存

-   **描述**：使用一个固定容量的缓存来存储 Agent 实例。当实例数量达到上限时，会自动淘汰最近最少被访问的实例。可以使用 `cachetools` 库的 `LRUCache` 快速实现。
-   **优点**：
    -   有效控制内存使用上限。
    -   保障活跃用户的性能体验。
    -   实现简单，无外部依赖。
-   **缺点**：
    -   服务重启后，所有内存中的 Agent 都会丢失。
    -   被淘汰的 Agent 状态会丢失，再次访问时需要重新创建。
    -   不适用于需要跨服务器实例共享会话的分布式部署。

### 方案三：TTL (Time To Live) 缓存

-   **描述**：为每个 Agent 实例设置一个固定的存活时间，例如30分钟。如果实例在这段时间内未被访问，则自动从内存中移除。
-   **优点**：
    -   可以确保“僵尸”实例最终被清理。
    -   实现简单，`cachetools` 库也提供 `TTLCache`。
-   **缺点**：
    -   策略较为粗暴，可能会淘汰掉一个正在思考但暂时没有操作的活跃用户，影响用户体验。通常来说，LRU 策略优于 TTL。

### 方案四：外部持久化存储 (分布式架构)

-   **描述**：这是大型互联网服务（如 Gemini, ChatGPT）采用的标准架构。服务本身变得“无状态”，Agent 的会话状态被存储在外部的持久化系统中。
-   **架构分层**：
    -   **无状态服务层 (Stateless Service Layer)**：负责处理业务逻辑和模型推理。自身不存储任何会话数据，可以随意扩缩容和重启。
    -   **热数据/缓存层 (Hot Data/Cache Layer)**：使用 **Redis** 等分布式内存数据库，存储活跃用户的短期对话历史（上下文），保证高频访问的性能。
    -   **冷数据/持久层 (Cold Data/Persistent Layer)**：使用 **PostgreSQL, DynamoDB, MongoDB** 等数据库，永久存储所有用户的完整对话历史。
-   **优点**：
    -   完美解决内存问题。
    -   用户状态永不丢失。
    -   支持服务横向扩展（分布式部署）。
    -   系统可用性和鲁棒性极高。
-   **缺点**：
    -   架构复杂，引入了 Redis、数据库等多个外部依赖，增加了开发和运维成本。

## 3. 最终决策与原因

**最终选型（当前阶段）**: **方案二：LRU (Least Recently Used) 缓存**。

**选择原因**:

1.  **解决核心痛点**: 以最低成本直接解决了当前最致命的“内存无限增长”问题，是保障服务稳定性的关键第一步。
2.  **实施成本低**: 无需引入新的外部服务依赖，仅需在代码层面进行少量修改，风险低、见效快。
3.  **体验与性能的平衡**: LRU 策略能有效区分活跃与非活跃用户，在保证活跃用户体验的同时，高效地回收了内存资源，是单体服务架构下的最佳实践。
4.  **保留未来扩展性**: 为项目未来的发展奠定了基础。当应用需要支持分布式部署或需要长期保存用户状态时，可以平滑地将 LRU 缓存模块替换为与外部持久化系统（如 Redis）交互的模块。

## 4. 实施建议

-   在 `KnowledgeAgentManager` 的 `__init__` 方法中，将 `self.agents` 初始化为 `cachetools.LRUCache`。
-   将缓存的 `maxsize` 参数提取到配置文件中 (`config/settings.py`)，例如 `AGENT_CACHE_MAX_SIZE`，方便在线调整，避免硬编码。
-   在 `pyproject.toml` 中添加 `cachetools` 作为项目依赖。
